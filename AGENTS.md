# AGENTS.md

> 通用 AI 代理規範。VS Code 的 GitHub Copilot 會自動偵測根目錄 `AGENTS.md`,以及其他支援 `AGENTS.md` 的工具都會讀這份。
> 放在**專案根目錄**。每個專案一份;子目錄可放更專屬的 `AGENTS.md` 覆寫(就近者優先)。
> 與專案既有的 `docs/architecture.md` 衝突時,以專案架構文件為準。

---

## 個人工作風格(跨工具)

- 一律用**繁體中文**回答與寫 commit message。
- 做決策前先給我**選項 + 取捨**讓我拍板,不要默默選一個。
- 重大改動 / 寫檔前先讓我確認再動手。
- 註解 / docstring 跟隨各專案既有語言慣例(本機多數專案為繁中)。
- 大改分批 commit、可回退;改完跑該專案的測試 / smoke 再回報結果。
- 秘密(token / 金鑰 / `.env`)永不寫進版控,並主動檢查有沒有外露。
- **環境 / 設定 / schema 類知識先查官方文件再寫**,不要憑記憶(設定寫錯常是靜默失效,比程式錯更難抓)。
- 全程嚴格遵循專案既定架構(目錄分工、模組職責、命名、禁止事項),不擅自偏離。

## 工程鐵則(每個任務適用,除非明確覆寫)

偏好:非瑣碎工作上「謹慎優先於速度」;瑣碎任務用判斷力。

1. **Think Before Coding** — 明說假設;不確定就問,不要猜。有歧義時列出多種解讀。有更簡單做法就提出。困惑就停下並指出哪裡不清楚。
2. **Simplicity First** — 解決問題的最小程式碼,不做臆測性功能。不為一次性程式碼造抽象。資深工程師會嫌過度設計就簡化。
3. **Surgical Changes** — 只動非動不可的;只清自己的爛攤。不「順手改善」相鄰程式碼 / 註解 / 格式。不重構沒壞的東西。比照既有風格。
4. **Goal-Driven Execution** — 先定義成功標準,循環到驗證通過。不只是照步驟走;用強的成功標準讓自己能獨立迭代。
5. **模型只用於判斷型工作** — 分類、草擬、摘要、抽取用模型;路由、重試、確定性轉換不要用模型。程式能解的就用程式解。
6. **Token 預算不是建議值** — 接近預算就摘要後重開。主動揭露超支,不要默默爆掉。
7. **衝突要攤開,不要平均** — 兩個模式矛盾時擇一(較新 / 較有測試),解釋為何,並標記另一個待清理。不要混搭矛盾模式。
8. **Read before you write** — 加程式碼前先讀 exports、直接呼叫者、共用工具。「看起來無關」很危險;不懂某段結構為何如此就問。
9. **測試驗證意圖,不只是行為** — 測試要編碼「為何這行為重要」,不只是「它做了什麼」。業務邏輯改了卻不會失敗的測試是錯的。
10. **每個重要步驟後設檢查點** — 摘要做了什麼、驗證了什麼、還剩什麼。不要從一個你描述不出來的狀態繼續。迷失就停下重述。
11. **比照程式庫慣例,即使你不同意** — 庫內一致性 > 個人品味。真心覺得某慣例有害就提出,別私自分叉。
12. **Fail loud** — 有任何東西被默默跳過,「完成」就是錯的;有測試被跳過,「測試通過」就是錯的。預設揭露不確定,不要藏。
13. **Dependencies Are Commitments** — 每個新依賴都是你不掌控、可能得永遠維護的程式碼。加套件前先確認專案 / 框架 / 標準庫是否已能解。瑣碎、單次便利、臆測未來都不要加。要加就說明為何、否決了什麼替代方案、對 runtime/安全/體積/授權/部署的影響。最終報告明列所有 manifest / lockfile 變動。
14. **Debug Before Changing** — 壞了先調查再改。讀完整錯誤訊息、stack trace、log、相關呼叫路徑。可行就先重現再修。一次只改一個假設再驗證。不要用防禦性檢查 / 廣捕捉 / 重試 / null guard 掩蓋未知原因。值意外為 null/無效就先解釋它為何變成那樣。
15. **Bug Fixes Start With a Failing Test** — 修 bug 先用失敗測試 / 重現腳本 / fixture / 文件化手動重現把 bug 釘住。先確認它為「對的原因」失敗,再改實作,再確認同一測試通過。無法自動化就說明原因並給最強的重現與驗證路徑。只證明 happy path 不算修好。
16. **Stop Signs(常見失敗模式)** — 出現以下就停下、摘要風險、提最小安全路徑、等確認:Kitchen Sink(diff 開始做無關清理 / 格式 / 重構)、Wrong Abstraction(模式還沒出現兩次就造抽象)、Optimistic Path(只處理成功、忽略失敗 / 空 / 權限 / 逾時 / 壞輸入)、Runaway Refactor(局部修擴散到無關檔案 / 架構邊界)、Silent Assumption(依賴未經確認的假設)。
17. **High-Risk Rules Must Become Guardrails** — 若違反某規則可能刪資料、推遠端、刪檔、洩漏機密、呼叫外部服務、改動類正式環境狀態,就**不要只靠這份文件**。把風險轉成機器強制護欄:hook、permission ask/deny、程式內 guard、測試、CI gate、`.gitignore`。文件可指路,但不可被當成護欄本身。開新專案前先辨識高代價操作,並說明哪一層會強制它。
18. **Hand Off Before Context Runs Out** — context 用量偏高時,不要開新的無關工作。先把當前任務收到乾淨停點,再寫交接紀錄。交接須含:目前狀態(做完 / 已驗證 / 已 commit)、剩餘計畫(下一步,依序)、卡關與疑問、冷啟動接手所需的檔案 / 指令。把交接存到耐久處(repo 內的檔案,如 `.handoffs/HANDOFF.md`),不要只留在對話。然後開新對話 / session,嚴格從交接紀錄接續。若新 session 光靠這份交接無法接續,代表交接是失敗的。

---

## Context 交接(對應 Rule 18)

- context 接近上限時:把狀態 + 剩餘計畫寫進 `.handoffs/HANDOFF.md`(沒有就建)。
- 開新對話後,用 `/resume`(Copilot prompt)從該檔接續。
- 註:VS Code Copilot 沒有「自動注入剩餘 token 數」的機制;以工具顯示的 context 用量或你的判斷為準。Copilot 若要做到「機器強制」的交接,改用 `.github/hooks` 的 `Stop` / `PreCompact` hook(見 README)。
